Getting Familiar with the SmartTool Library

 

Introduction

The PowerBuilder setup program installs SmartTool.PBL in the PowerBuilder Code Examples\MobiLink\SmartTool directory. You can add this library to any of your PowerBuilder targets that use MobiLink synchronization. The SmartTool library contains the following objects that you can use to enhance your MobiLink smart client applications:

 

  Windows:    w_login_prompt

                       w_prompt_new_dbpassword

                       w_prompt_new_mlpassword

 

   Global Functions:    gf_get_syncsettings_from_db

                                    gf_prompt_new_dbpassword

                                    gf_ prompt_new_dbmlssword

                                    gf_save_syncsettings

   

    Structure:     s_new_password

 

The SmartTool toolkit also includes the SQL file, SyncSettings.sql, that adds a table to the remote databases for your smart client applications.

 

This document describes each of the objects in the SmartTool toolkit and describes how to incorporate these objects into your application. The SmartTool library objects supplement the objects generated by the MobiLink Synchronization wizard in PowerBuilder.

w_login_prompt.srw

Keeping the remote database secure is critical for smart client applications, because sensitive corporate data resides in the remote database.  Applications should require authentication by prompting the user to supply both a database password and an encryption key at runtime.  The login window assists by generating a complete ODBC ConnectString based on values supplied by the user at runtime.

 

 

 

Currently, the w_login_prompt response window is designed to support ODBC connections only.  It is called with a partially complete ODBC transaction object as an input parameter.  

 

It begins by parsing the keyword=value pairs that make up the incoming  ConnectString, and keeps these values in a number of instance variables.  Then it prompts the user to supply password and encryption key values.  When the user clicks OK, it reconstructs the ConnectString, adding the new PWD and DBKEY keyword=value pairs.

 

SyncSettings.sql

The MLSync.GetSyncRegistryProperties() and MLSync.SetSyncRegistryProperties() functions allow users to change MLSync properties at runtime and then store these changes in the Windows Registry for reuse.  The w_<appname>_sync_options window that is generated by the MobiLink Synchronization Wizard performs this task.   However, for highly sensitive properties such as dbpass, mlpass, encryptionkey, and authenticationparms, this approach is insufficient.  Either the end user must enter these values manually each time synchronization is run, or must store these properties in a more secure location. 

 

SyncSettings.sql helps you to implement the latter solution by creating a SyncSettings table in the remote database itself.  A SmartTool toolkit function, gf_save_syncsettings(), stores a row in this table and another function, gf_get_sync_settings_from_db(), retrieves the contents of a row.  Each row of the table contains the same properties as a PowerBuilder SyncParm structure.

 

create table SyncSettings (

     regkey    varchar(250) not null primary key,

     dbuser    varchar(128) not null,

     dbpass    varchar(128) not null,

     mluser    varchar(128) not null,

     mlpass    varchar(128),

     encrypt   varchar(128),

     authparms varchar(128))

Installation:

To install the SyncSettings table in your remote database, create a database profile for your remote database and connect to the database from the PowerBuilder Database painter.  In the ISQL View, choose the “Edit, Paste Special, From File…” menu item and choose SyncSettings.sql.  Then choose “Design, Execute ISQL…”. 

 

Alternately, issue the following command from a CMD prompt:

"%ASA_PATH%\win32\dbisql" -q -nogui -c "uid=dba;pwd=sql;dbkey=secret;dsn=SalesDB_remote" -ODBC read syncsettings.sql

Usage:

The MobiLink Synchronization wizard generates a function called gf_<appname>_configure_sync().  Once the syncsettings table is installed in your remote database, you can uncomment the PowerScript code that calls gf_get_sync_settings_from_db() and gf_save_syncsettings().

 

gf_get_syncsettings_from_db(sqlca, &

l_uosync.syncregistrykey, parms)

l_uosync.SetParm(parms) 

OpenWithParm(w_salesdb_sync_options, l_uosync)

     parms = Message.PowerObjectParm

     if parms.returncode = 0 then

gf_save_syncsettings(sqlca, l_uosync.syncregistrykey, parms)

          …

end if

gf_get_sync_settings_from_db.srf

This function retrieves a row in the SyncSettings table and populates a SyncParm structure with its contents.

Arguments:

Datatype

Argument

Description

Transaction

A_trans

The transaction object currently connected to the SQL Anywhere remote database that contains the dbo.syncsettings table

String

A_regkey

This MLSync.SyncRegistryKey value is used as a primary key value in the SyncSettings table to uniquely identify the row containing the passwords and authentication parms. 

SyncParm

A_parms

This structure is populated with the contents of the row (in the SyncSettings table) that is indexed by the A_regkey argument.  

Return Code:

After the retrieve statement is issued, the function return code is set to A_trans.sqlcode.

Value

Description

0

Function succeeded

100

Row not found

-1

An error occurred

 

Usage:

gf_get_syncsettings_from_db(sqlca, &

l_uosync.syncregistrykey, parms)

l_uosync.SetParm(parms) 

OpenWithParm(w_salesdb_sync_options, l_uosync)

 

When w_salesdb_sync_options opens, the fields for dbuser, dbpass, mluser, mlpass, encryptionkey, and authenticationparms will already be populated with the values stored in the SyncSettings table. The user does not have to re-enter these properties each time synchronization occurs. However, the properties always remain secure, and the passwords can even be updated by the end user.

gf_prompt_new_dbpassword.srf

This function allows the application to easily change a database password associated with a user ID in the SQL Anywhere remote database.  The function also updates the dbpass property in the MLSync object and may optionally call gf_save_syncsettings() to save the   new database password in the syncsettings table.  This ensures that the next time the user runs the MLSync.synchronize function, the new database password will be used.

 

Users with mobile devices should be encouraged to change their database passwords regularly.

Usage:

Add a new menu item entitled “Change DB Password…” and insert this code in the clicked! event:

 

MLSync        l_uosync

long          rc

string        errtext

l_uosync = CREATE nvo_salesdb_mlsync

if IsValid(l_uosync) then

     rc = gf_prompt_new_dbpassword(sqlca, l_uosync, errtext)

     messagebox("Change DB Password", errtext)

end if

destroy l_uosync

w_prompt_new_dbpassword.srw

This response window is launched by gf_prompt_new_dbpassword and takes an instance of s_new_password (i_newpass) as a parameter.  The open event uses the i_newpass.uosync.dbuser and i_newpass.uosync.dbpass properties to initialize the first two single line edit controls and prompts the user to enter a new password twice.  The OK button is enabled when sle_newpass2 loses focus and the new password is verified.   When OK is clicked, the new password value is returned to the caller via s_new_password.newpassword.

 

 

 

The closequery() event sends i_newpass structure back to the caller via Message.PowerObjectParm.
gf_prompt_new_mlpassword.srf

This function allows the application to easily change a MobiLink password which is stored on the consolidated database.  The function also updates the mlpass property in the MLSync object and optionally calls gf_save_syncsettings() to save the new MobiLink password in the SyncSettings table. This ensures that the next time the user runs the MLSync.synchronize function, the new password will be used.

 

Users with mobile devices should be encouraged to change their MobiLink passwords regularly.
